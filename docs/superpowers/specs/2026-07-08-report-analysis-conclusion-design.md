# 巡检报告分析结论设计

| 项目 | 值 |
|------|----|
| 日期 | 2026-07-08 |
| 版本 | v1.0 |
| 状态 | 待评审 |
| 影响范围 | `report.sh` 报告生成逻辑 |

## 1. 背景

当前报告已经包含巡检概览、问题汇总和各章节明细，但读者需要自行从表格和问题列表中归纳结论。为了让报告更适合交付和复核，需要增加“分析结论”：

1. **总体结论**：放在第二章“问题汇总”内，汇总本次巡检的整体状态，并列出所有警告项。
2. **章节结论**：每个 3.x 巡检章节末尾增加本节小结，用模板化自然语言说明该章节是否存在异常。

本需求不引入复杂归因，不推断业务影响，只根据已有字段和阈值判断结果生成可读结论。

## 2. 目标

- 在 `# 2. 问题汇总` 下新增 `## 2.1 总体结论`
- 原有 `2.1 严重问题`、`2.2 警告项`、`2.3 建议优化项` 顺延为 `2.2`、`2.3`、`2.4`
- 总体结论汇总服务器健康状态、严重问题数、警告项数、建议项数，并列出所有警告项
- 在 `3.1` 到 `3.7` 每个章节末尾增加“本节小结”
- 章节小结使用模板化自然语言，包含整体判断、异常明细和轻量处置建议
- 不修改 `collect.sh`，不新增采集字段，不新增外部依赖

## 3. 非目标

- 不实现 AI 式复杂归因，例如“由于业务流量增长导致磁盘上涨”
- 不生成趋势分析，因为当前没有历史数据输入
- 不改变 Word 样式、字体、颜色或封面排版
- 不改变 `.dat` 键值格式
- 不把建议项扩展成自动化处置方案

## 4. 报告结构调整

调整后的第二章结构：

```markdown
# 2. 问题汇总

## 2.1 总体结论

## 2.2 严重问题（需立即处理）

## 2.3 警告项（建议处理）

## 2.4 建议优化项
```

各应用巡检章节末尾增加小结：

```markdown
## 3.1 资源使用概况
...
### 3.1.5 本节小结

## 3.2 网络状态
...
### 3.2.5 本节小结

## 3.3 进程与 Java 应用
...
### 3.3.5 本节小结
```

后续章节以现有子章节数量为准，新增小结编号放在本章最后。

## 5. 总体结论规则

总体结论基于 `SERVER_HEALTH`、`ISSUES_CRIT`、`ISSUES_WARN`、`ISSUES_INFO` 生成。

### 5.1 无问题模板

```text
本次共巡检 N 台服务器，巡检结果整体平稳，未发现严重问题和警告项。建议按常规周期持续观察。
```

### 5.2 存在警告模板

```text
本次共巡检 N 台服务器，其中 X 台正常、Y 台存在警告、Z 台存在异常。共发现严重问题 A 项、警告项 B 项、建议优化项 C 项。当前主要需要关注以下警告项：

- [host] ...
- [host] ...
```

### 5.3 存在严重问题模板

```text
本次共巡检 N 台服务器，其中 X 台正常、Y 台存在警告、Z 台存在异常。共发现严重问题 A 项、警告项 B 项、建议优化项 C 项。报告中存在需要优先处理的严重问题，建议先处理严重问题，再复核警告项。

当前警告项如下：
- [host] ...
```

如果没有警告项，则输出：

```text
本次未发现警告项。
```

## 6. 章节小结规则

章节小结使用“整体判断 + 异常明细 + 轻量建议”的模板，不做复杂原因分析。

### 6.1 正常模板

可按章节选择不同措辞，避免所有小结完全重复：

```text
本节资源使用情况整体平稳，各服务器 CPU、内存、磁盘与 IO 指标均未触发告警阈值。
```

```text
本节网络状态整体正常，监听端口、TCP 连接和网络附属检查未发现明显异常。
```

```text
本节检查项整体正常，未发现超过预设阈值的异常指标。
```

### 6.2 警告模板

```text
本节整体状态可接受，但发现部分指标触发警告阈值，建议结合业务负载持续观察：host 的 metric 为 value。
```

### 6.3 严重异常模板

```text
本节发现需要优先处理的异常项：host 的 metric 为 value，已超过严重阈值，建议尽快复核并处理。
```

### 6.4 混合模板

```text
本节大部分检查项处于合理范围，但发现以下需关注项：host 的 metric 为 value；host2 的 metric2 为 value2。
```

如果异常明细过多，最多展示前 5 条，并补充：

```text
其余 N 项请查看本章明细表。
```

## 7. 章节数据来源

| 章节 | 小结数据来源 | 结论重点 |
|------|--------------|----------|
| 3.1 资源使用概况 | CPU、内存、SWAP、磁盘、IO 阈值判断 | 资源是否超阈值 |
| 3.2 网络状态 | TCP 状态阈值、网络字段可用性 | 网络连接是否存在异常 |
| 3.3 进程与 Java 应用 | 僵尸进程、服务状态、Java OOM/GC 字段 | 进程和 Java 应用是否异常 |
| 3.4 中间件状态 | Redis/Nacos/MySQL 运行状态 | 中间件是否运行 |
| 3.5 系统安全 | SSH、安全组件、NTP、登录失败 | 安全配置是否有风险 |
| 3.6 Crontab 分析 | `CRONTAB_ANALYSIS` 和任务条目 | 定时任务是否存在异常 |
| 3.7 日志与告警 | OOM、内核错误、安全告警、系统日志错误 | 日志告警是否需要关注 |

## 8. 实现策略

在 `report.sh` 中新增章节级结论收集结构，避免从最终 Markdown 反向解析。

推荐新增：

```bash
declare -A SECTION_CRIT=()
declare -A SECTION_WARN=()
declare -A SECTION_INFO=()
```

新增 helper：

```bash
add_section_finding section level message
generate_overall_conclusion
generate_section_conclusion section title normal_text
```

各章节生成表格时，沿用现有阈值判断逻辑，同时调用 `add_section_finding` 记录本节结论需要的自然语言片段。问题汇总继续使用现有 `add_issue`。
